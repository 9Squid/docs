Every request to the Buildmarkets API must include a valid API key pair. There are no sessions, cookies, or OAuth flows — each request is independently authenticated by two HTTP headers that identify your partner tenant and authorize the action being performed.

## Required headers

Include both headers on every request:

```bash
curl https://dev-tapp-api.tappengine.com/v1/investors \
  -H "X-API-Key: YOUR_API_KEY" \
  -H "X-API-Secret: YOUR_API_SECRET"
```

If either header is missing, invalid, or belongs to a revoked key, the API returns `401 Unauthorized`. If the key is valid but lacks the required scope for the endpoint, the API returns `403 Forbidden`.

## Key scopes

When you generate a key, you assign it one or more **scopes** that control exactly which endpoints it can call. Follow the principle of least privilege: assign only the scopes your service actually needs.

| Scope | Access granted |
|---|---|
| `accounts:read` | Read investor details, balances, and positions |
| `accounts:write` | Create, update, and close investor accounts |
| `funding:read` | List funding sources and view deposit/withdrawal history |
| `funding:write` | Connect funding sources, initiate deposits and withdrawals |
| `trading:read` | Read orders and executions |
| `trading:write` | Place, modify, and cancel orders |
| `marketdata:read` | Access asset search, quotes, and market data endpoints |
| `documents:read` | List and retrieve account documents |
| `documents:write` | Upload compliance documents |
| `webhooks:read` | List webhook configurations |
| `webhooks:write` | Create, update, and delete webhooks |
| `keys:manage` | Generate and revoke API keys programmatically |

## Generate keys via the dashboard

Log into the Buildmarkets Partner Dashboard, navigate to **Settings → API Keys**, and click **+ Generate Key**. Give the key a descriptive label — for example, `backend-prod` or `sandbox-integration-test` — and select the scopes it needs.

## Generate keys via the API

If you have an existing key with the `keys:manage` scope, you can generate new keys programmatically:

```bash
curl -X POST https://dev-tapp-api.tappengine.com/v1/keys \
  -H "X-API-Key: YOUR_EXISTING_KEY" \
  -H "X-API-Secret: YOUR_EXISTING_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "label": "backend-service-prod",
    "scopes": ["accounts:read", "accounts:write", "trading:write", "marketdata:read"]
  }'
```

**Response:**

```json
{
  "id": "key_01j9x8m2k7qpzwv3t5r6y8n0ab",
  "label": "backend-service-prod",
  "api_key": "bm_key_abc123xyz",
  "api_secret": "bm_secret_xyz789abc",
  "scopes": ["accounts:read", "accounts:write", "trading:write", "marketdata:read"],
  "created_at": "2025-06-01T12:00:00.000Z"
}
```

<Warning>
  The `api_secret` is only returned once — at creation time. Store it immediately in a secrets manager (such as AWS Secrets Manager, GCP Secret Manager, or HashiCorp Vault). It cannot be retrieved again. If you lose it, revoke the key and generate a new one.
</Warning>

## List your API keys

Retrieve metadata for all keys on your partner account. Secret values are never included in list responses.

```bash
curl https://dev-tapp-api.tappengine.com/v1/keys \
  -H "X-API-Key: YOUR_API_KEY" \
  -H "X-API-Secret: YOUR_API_SECRET"
```

## Revoke a key

Revoke a key that is no longer needed or that you suspect has been compromised:

```bash
curl -X DELETE https://dev-tapp-api.tappengine.com/v1/keys/KEY_ID \
  -H "X-API-Key: YOUR_API_KEY" \
  -H "X-API-Secret: YOUR_API_SECRET"
```

Returns `204 No Content` on success. Revocation is immediate — any subsequent request using the revoked key returns `401 Unauthorized`.

## Error responses

| HTTP status | Meaning |
|---|---|
| `401 Unauthorized` | Missing, invalid, or revoked `X-API-Key` or `X-API-Secret` |
| `403 Forbidden` | Key is valid but lacks the required scope for this endpoint |

## Security best practices

| Practice | Guidance |
|---|---|
| **Never expose keys client-side** | Use API keys only in server-side code. Never embed them in mobile app bundles, browser JavaScript, or public repositories. |
| **Use separate keys per environment** | Maintain distinct key pairs for sandbox and production. A sandbox key should never be used against the production URL. |
| **Rotate keys regularly** | Generate a replacement key, update your service configuration, then revoke the old key. Aim to rotate at least every 90 days. |
| **Scope keys minimally** | A backend service that only places orders should have `trading:write` — not `accounts:write` or `keys:manage`. |
| **Store secrets securely** | Use a dedicated secrets manager. Do not store secrets in `.env` files committed to source control or in plaintext configuration. |
| **Monitor for anomalies** | Review API key activity in the Partner Dashboard. Set up alerts on unexpected request volume spikes or error-rate increases. |
| **Revoke immediately on compromise** | If a key is accidentally exposed in a log, a repository, or a deployment, revoke it before doing anything else. |
