Every error response from the Buildmarkets API uses the same JSON envelope, regardless of whether the problem is a missing field, an authentication failure, a business-rule violation, or a server fault. Understanding this structure lets you write resilient integration code that branches on specific error types, surfaces meaningful messages to your users, and retries only the requests that are safe to retry.

---

## Error response envelope

All error responses carry a top-level `error` object with four fields:

```json
{
  "error": {
    "type": "invalid_request_error",
    "code": "BLUM-400-002",
    "message": "Required field 'symbol' is missing.",
    "param": "symbol"
  }
}
```

<ResponseField name="error.type" type="string">
  A broad category string you can use to branch your error-handling logic. See the [error types table](#error-types) below.
</ResponseField>

<ResponseField name="error.code" type="string">
  A specific `BLUM-{STATUS}-{SEQ}` code that pinpoints the exact error condition. Use this for programmatic handling and when filing support requests.
</ResponseField>

<ResponseField name="error.message" type="string">
  A developer-readable explanation of what went wrong. Do not display this string directly to end users — map it to your own UI copy.
</ResponseField>

<ResponseField name="error.param" type="string">
  Present when the error relates to a specific request field. Contains the dot-notation path to the offending parameter (e.g. `tax_id`, `address.postal_code`).
</ResponseField>

### BLUM error code format

Every error code follows the pattern `BLUM-{HTTP_STATUS}-{SEQ}`:

- **`BLUM-400-002`** — bad request, sequence 002
- **`BLUM-401-001`** — authentication error, sequence 001
- **`BLUM-404-001`** — not found, sequence 001

Always log the full code. When you contact Buildmarkets support, include the `code` value and the request body so the support team can trace the failure.

## HTTP status codes

### 2xx — Success

| Code | Meaning |
|---|---|
| `200 OK` | Request succeeded. Response body contains the resource. |
| `201 Created` | Resource was created. Response body contains the new resource. |
| `204 No Content` | Request succeeded with no response body (e.g. DELETE operations). |

### 4xx — Client errors

These errors mean your request was the problem. Fixing the request resolves the error — retrying without changes will not help.

| Code | Meaning | Common causes |
|---|---|---|
| `400 Bad Request` | The request body or parameters are invalid. | Missing required fields, unrecognised enum values, malformed JSON |
| `401 Unauthorized` | Authentication failed. | Missing, invalid, or expired `X-API-Key` / `X-API-Secret` headers |
| `403 Forbidden` | Authenticated but not authorised. | API key lacks the required permission scope for this endpoint |
| `404 Not Found` | The requested resource does not exist. | Invalid `investor_id`, `order_id`, or other path parameter |
| `409 Conflict` | Request conflicts with the current resource state. | Duplicate `client_order_id`, account already closed |
| `422 Unprocessable Entity` | Request is valid but fails a business rule. | KYC not yet approved, insufficient balance for withdrawal |
| `429 Too Many Requests` | Rate limit exceeded. | Too many requests per second or per minute — see [rate limiting](#rate-limiting) below |

### 5xx — Server errors

These errors originate on the Buildmarkets side. They are generally safe to retry with backoff.

| Code | Meaning |
|---|---|
| `500 Internal Server Error` | An unexpected fault occurred on Buildmarkets's infrastructure. |
| `503 Service Unavailable` | A Buildmarkets service or upstream dependency is temporarily unavailable. |

## Error types

The `error.type` field maps to one of the following categories. Use this field to drive high-level branching in your error handler before inspecting the specific `code`.

| Type | HTTP status(es) | Description |
|---|---|---|
| `invalid_request_error` | 400, 422 | The request body or parameters failed validation or a business-rule check. |
| `authentication_error` | 401 | The API key or secret is missing, invalid, or has been revoked. |
| `permission_error` | 403 | The authenticated key does not have access to this endpoint or resource. |
| `not_found_error` | 404 | The requested resource could not be found. |
| `conflict_error` | 409 | The request conflicts with the current state of the resource (e.g. duplicate key). |
| `idempotency_error` | 409 | A request was replayed with the same idempotency key but a different body. |
| `rate_limit_error` | 429 | The request was rejected because you have exceeded the rate limit. |
| `api_error` | 500, 503 | An unexpected error occurred on Buildmarkets's side. |

## Rate limiting

Buildmarkets enforces per-key rate limits to ensure platform stability for all partners.

| Window | Default limit |
|---|---|
| Per second (per API key) | 10 requests |
| Per minute (per API key) | 300 requests |
| Per day (per partner) | Contact your Buildmarkets representative |

When you exceed the limit, you receive a `429 Too Many Requests` response. The response includes a `Retry-After` header telling you how many seconds to wait before your next request:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 2

{
  "error": {
    "type": "rate_limit_error",
    "code": "BLUM-429-001",
    "message": "Rate limit exceeded. Please retry after 2 seconds."
  }
}
```

Implement **exponential backoff** in your retry logic — do not immediately hammer the API after a `429`. A safe strategy: wait for the `Retry-After` duration, then retry. If you receive consecutive `429`s, double your wait time on each attempt.

---

## Retryable vs. non-retryable errors

| HTTP status | Retry? | Guidance |
|---|---|---|
| `400` | No | Fix the request — the error is deterministic and will not resolve on its own. |
| `401` | No | Check your `X-API-Key` and `X-API-Secret` values. |
| `403` | No | The key lacks the required scope — contact your Buildmarkets representative. |
| `404` | No | The resource does not exist — verify the ID in your request path. |
| `409` | No | Check whether a duplicate already exists before creating a new resource. |
| `422` | No | A business-rule blocked the request — resolve the underlying condition first. |
| `429` | Yes | Wait for `Retry-After` seconds, then retry with exponential backoff. |
| `500` | Yes | Retry with exponential backoff (1 s → 2 s → 4 s → 8 s). Alert your team after 3–5 failed retries. |
| `503` | Yes | Same as `500`. Indicates a temporary service disruption. |

## Handling errors in practice

Use `error.type` for high-level branching and `error.code` for specific cases. The following curl command deliberately omits the required `symbol` field so you can see a real `400` error response in your terminal:

```bash
curl -X POST https://dev-tapp-api.tappengine.com/v1/investors/inv_01j9x8m2k7qpzwv3t5r6y8n0ab/orders \
  -H "X-API-Key: $BUILDMARKETS_API_KEY" \
  -H "X-API-Secret: $BUILDMARKETS_API_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "class": "equity",
    "type": "market",
    "side": "buy",
    "quantity": "10",
    "time_in_force": "day"
  }'
```

Expected response:

```json
{
  "error": {
    "type": "invalid_request_error",
    "code": "BLUM-400-002",
    "message": "Required field 'symbol' is missing.",
    "param": "symbol"
  }
}
```

Handle errors programmatically by switching on `error.type` and `error.code`:

```javascript
const response = await fetch('https://api.tappengine.com/v1/investors/inv_123/orders', {
  method: 'POST',
  headers: {
    'X-API-Key': process.env.BUILDMARKETS_API_KEY,
    'X-API-Secret': process.env.BUILDMARKETS_API_SECRET,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify(orderPayload),
});

if (!response.ok) {
  const { error } = await response.json();

  switch (error.type) {
    case 'authentication_error':
      // Credentials are wrong — alert your ops team immediately
      alertOpsTeam('Buildmarkets auth failure', error.code);
      break;

    case 'invalid_request_error':
      if (error.code === 'BLUM-409-002') {
        // Duplicate client_order_id — this may be a retry hitting an existing order
        // Fetch the existing order by client_order_id instead of creating a new one
        return fetchOrderByClientId(orderPayload.client_order_id);
      }
      // Other validation errors — surface to the calling layer
      throw new ValidationError(error.message, error.param);

    case 'rate_limit_error':
      // Back off and retry
      await sleep(parseInt(response.headers.get('Retry-After') || '2', 10) * 1000);
      return retryRequest();

    case 'api_error':
      // Server-side fault — log and retry with backoff
      logger.error('Buildmarkets server error', { code: error.code });
      return retryWithBackoff();

    default:
      throw new Error(`Unhandled Buildmarkets error: ${error.code} — ${error.message}`);
  }
}
```

<Tip>
  Always set `client_order_id` on every order creation request. If your request times out and you retry, a `409` with code `BLUM-409-002` tells you the original order already exists. You can then safely fetch it by `client_order_id` instead of accidentally placing a duplicate order. See [Idempotency](/platform/idempotency) for the full pattern.
</Tip>
