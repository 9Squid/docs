The Buildmarkets sandbox is a full-fidelity replica of the production environment. Every endpoint, request shape, and response schema is identical — the only differences are that KYC is simulated using test SSN values, orders fill without touching real markets, and ACH transfers move no actual money. Build your entire integration in sandbox before you touch a single production credential.

<Note>
  All sandbox activity is completely isolated from production. Accounts, orders, deposits, and API keys created in sandbox have no effect on production data whatsoever.
</Note>

## Sandbox base URL

```
https://dev-tapp-api.tappengine.com
```

Use this as the base for every API call during development and QA. When you're ready to go live, swap this URL for the production base URL — no other changes to your code are required.

## Authentication in sandbox

Sandbox uses the same `X-API-Key` / `X-API-Secret` header authentication as production. Generate sandbox keys from **Settings → API Keys** in the Partner Dashboard, and clearly label them (e.g., `sandbox-dev`) so you never confuse them with production keys.

```bash
curl https://dev-tapp-api.tappengine.com/v1/investors \
  -H "X-API-Key: YOUR_SANDBOX_KEY" \
  -H "X-API-Secret: YOUR_SANDBOX_SECRET"
```

## Simulated KYC outcomes

In sandbox, KYC is not submitted to a real identity verification provider. Instead, the outcome is determined entirely by the last four digits of the SSN you provide when creating an investor. This lets you test every compliance code path in your application without waiting for real verification results.

| SSN suffix | KYC outcome | Account status after check |
|---|---|---|
| `0001` | `APPROVED` | `active` — ready for funding and trading |
| `0002` | `REJECTED` | `closed` — account is declined |
| `0003` | `MANUAL_REVIEW` | `approval_pending` — account awaits manual review |

**Example:** To create an investor who will be instantly approved, use any SSN ending in `0001`, such as `111-22-0001` or `987-65-0001`. The middle segments can be anything valid.

```bash
# This investor will be APPROVED
curl -X POST https://dev-tapp-api.tappengine.com/v1/investors \
  -H "X-API-Key: $BM_KEY" \
  -H "X-API-Secret: $BM_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "account_type": "individual",
    "management_type": "self_directed",
    "first_name": "Test",
    "last_name": "User",
    "tax_id": "111-22-0001",
    "tax_id_type": "SSN",
    ...
  }'
```

Test all three outcomes in your application before going live. Your UI should handle rejected accounts gracefully and surface a clear message for accounts in manual review.

## Simulated order fills

Orders placed in sandbox do not route to real markets. They fill automatically with simulated execution prices. You can use this to:

- Verify your order placement logic and `client_order_id` idempotency handling
- Test your webhook handler for `order.filled` and `order.canceled` events
- Confirm that positions update correctly after a fill
- Exercise your rebalancing logic end-to-end

All order types are supported in sandbox: `market`, `limit`, `stop`, `stop_limit`, and `trailing_stop`. Notional orders (using `notional` instead of `quantity`) also work as expected.

## Simulated ACH transfers

ACH deposits and withdrawals in sandbox do not initiate real bank transfers. They immediately transition to a settled state so you can proceed with funding-dependent test scenarios without waiting for clearing windows.

Use this to:

- Verify your deposit initiation flow and webhook handling for `transfer.deposit.completed`
- Test the full end-to-end path from bank link → deposit → order placement → position check
- Confirm your error handling for edge cases like insufficient balance on withdrawal

## Sandbox vs production differences

| Attribute | Sandbox | Production |
|---|---|---|
| **Base URL** | `https://dev-tapp-api.tappengine.com` | `https://api.tappengine.com` |
| **KYC verification** | Simulated via SSN suffix | Real identity verification |
| **Order routing** | Simulated fills, no real execution | Live order routing to markets |
| **ACH settlement** | Immediate (no wait) | 1–3 business days |
| **Real money** | Never — no real funds involved | Yes — real customer funds |
| **API keys** | Sandbox-specific key pairs | Production-specific key pairs |
| **Data isolation** | Fully isolated from production | Live data only |
| **Rate limits** | Relaxed (suitable for load testing) | Production rate limits apply |

## Switching from sandbox to production

Switching environments requires only two changes:

1. **Replace the base URL** — swap `https://dev-tapp-api.tappengine.com` with `https://api.tappengine.com`
2. **Replace your API credentials** — use your production `X-API-Key` and `X-API-Secret`, not your sandbox keys

No endpoint paths, request bodies, or response handling code needs to change. This is intentional — the sandbox is an exact structural mirror of production.

```bash
# Sandbox
curl https://dev-tapp-api.tappengine.com/v1/investors \
  -H "X-API-Key: SANDBOX_KEY" \
  -H "X-API-Secret: SANDBOX_SECRET"

# Production — only the URL and credentials change
curl https://api.tappengine.com/v1/investors \
  -H "X-API-Key: PRODUCTION_KEY" \
  -H "X-API-Secret: PRODUCTION_SECRET"
```

<Warning>
  Never use sandbox API keys in a production environment, and never send real customer data (real SSNs, real bank account numbers) to the sandbox URL. Keep the two environments entirely separate throughout your development and operations lifecycle.
</Warning>

## Recommended sandbox test scenarios

Before requesting production access, cover these scenarios in sandbox:

- Create an investor with SSN suffix `0001` → verify `active` status
- Create an investor with SSN suffix `0002` → verify your UI handles rejection gracefully
- Create an investor with SSN suffix `0003` → verify your UI surfaces a "pending review" state
- Link a bank account via Plaid Link test credentials → verify `funding_source_id` is returned
- Initiate a deposit → verify `transfer.deposit.completed` webhook fires
- Place a market buy order → verify `order.filled` webhook fires and position appears
- Place an order with the same `client_order_id` twice → verify only one order is created
- Cancel an open order → verify status transitions to `cancelled`
- Attempt an order with insufficient balance → verify your error handling for `400` responses

Completing this checklist gives you high confidence that your integration will behave correctly in production.
