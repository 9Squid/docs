Buildmarkets verifies every applicant's identity documents through its KYC process before the account is booked on the clearing system. Both US applicants (`tax_id_type` of `USA_SSN` / `USA_EIN`) and non-US applicants (`tax_id_type` of `FIN`) go through the same document-based identity verification.

You simply submit the government ID image(s) with the account application, and Buildmarkets orchestrates the identity review and clearing-system submission behind the scenes.

---

## How it works

1. **Submit documents with the application.** Include the `documents` array in `POST /v1/accounts` (see [Account Opening](/guides/self-directed/account-onboarding)). The account is created immediately with status `SUBMITTED` and `kyc_status: "submitted"`; `account_number` is `null` at this point.
2. **Identity review.** The document images go through KYC verification. Most reviews complete in seconds to minutes.
3. **Clearing-system booking.** When the identity verification is approved, Buildmarkets automatically submits the application to the clearing system, which runs CIP screening (KYC \+ OFAC watchlist) and assigns the account number. The account moves through `APPROVAL_PENDING` to `ACTIVE`.
4. **Rejection or manual review.** If verification is rejected or needs more information, the account moves to `ACTION_REQUIRED` (`kyc_status: "action_required"` or `"rejected"`).

## Submitting identity documents

Documents are passed in the `documents` array of the account-creation request. Each entry is one base64-encoded image:

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `document_type` | string |  | Defaults to `identity_verification` |
| `document_sub_type` | string | ✅ | `passport`, `id_card`, or `drivers_license` |
| `side` | string |  | `front` or `back` — required for two-sided IDs |
| `content` | string | ✅ | Base64-encoded image data |
| `mime_type` | string |  | Defaults to `image/jpeg` |

**Image requirements:**

- A `passport` requires **one** entry.
- An `id_card` or `drivers_license` requires **two** entries: one with `side: "front"` and one with `side: "back"`.
- `content` must be valid base64 image data — placeholder or malformed content is rejected with a `400` before the application is submitted.

### Example — account creation with a driver's license

```bash
FRONT=$(base64 -i license_front.jpg)
BACK=$(base64 -i license_back.jpg)

curl -X POST https://api.tappengine.com/v1/accounts \
  -H "X-API-Key: $BUILDMARKETS_API_KEY" -H "X-API-Secret: $BUILDMARKETS_API_SECRET" \
  -H "Content-Type: application/json" \
  -d "{
    \"account_type\": \"individual\",
    \"contact\": { \"email_address\": \"jane@example.com\", \"city\": \"Austin\", \"state\": \"TX\", \"postal_code\": \"78701\" },
    \"identity\": { \"given_name\": \"Jane\", \"family_name\": \"Doe\", \"date_of_birth\": \"1990-04-15\", \"tax_id\": \"111-22-3344\", \"tax_id_type\": \"USA_SSN\" },
    \"investment_profile\": { \"risk_tolerance\": \"moderate\", \"investment_objective\": \"growth\", \"annual_income_range\": \"50000-100000\", \"net_worth_range\": \"100000-250000\", \"liquid_net_worth_range\": \"50000-100000\" },
    \"agreements\": [ { \"agreement\": \"customer_agreement\", \"signed_at\": \"2026-06-01T12:00:00Z\" } ],
    \"documents\": [
      { \"document_sub_type\": \"drivers_license\", \"side\": \"front\", \"content\": \"$FRONT\", \"mime_type\": \"image/jpeg\" },
      { \"document_sub_type\": \"drivers_license\", \"side\": \"back\", \"content\": \"$BACK\", \"mime_type\": \"image/jpeg\" }
    ]
  }"
```

### Non-US applicants (deferred flow)

For non-US applicants (`tax_id_type: "FIN"`), the document review is fully deferred: the application is held until the identity review approves, and only then is the account booked on the clearing system. No additional API calls are needed — the same `POST /v1/accounts` request with the `documents` array covers it, and status changes arrive via webhooks.

## Tracking verification outcomes

Watch the account's `status` and `kyc_status` fields:

| `kyc_status` | Meaning |
| --- | --- |
| `submitted` | Identity data awaiting processing |
| `approved` | Identity verified |
| `rejected` | Identity could not be verified |
| `action_required` | Requires additional action — either customer resubmission or compliance review; check `kyc_results` for the reason |

The account response also includes a `kyc_results` object with the verification outcome: `summary` (`pass`, `fail`, or `indeterminate`), `reject` (a list of rejection reason labels), and `additional_information`. It is `null` while the review is still in progress.

Subscribe to the `account.updated` webhook event for real-time updates, or poll `GET /v1/accounts/{accountId}`.

## Resubmitting documents

If verification is rejected or could not complete (account in `ACTION_REQUIRED`), submit new document images via `PATCH /v1/accounts/{accountId}` with a fresh `documents` array — this re-runs identity verification with the new image(s):

```bash
curl -X PATCH https://api.tappengine.com/v1/accounts/b3d9f1a2-4c7e-4f0b-9e2d-1a2b3c4d5e6f \
  -H "X-API-Key: $BUILDMARKETS_API_KEY" -H "X-API-Secret: $BUILDMARKETS_API_SECRET" \
  -H "Content-Type: application/json" \
  -d "{
    \"documents\": [
      { \"document_sub_type\": \"passport\", \"content\": \"$(base64 -i passport_scan.jpg)\", \"mime_type\": \"image/jpeg\" }
    ]
  }"
```

## Verified users and additional accounts

Document verification happens **once per user**, not once per account. When any of a user's accounts passes KYC, the user is marked KYC-approved. Additional accounts opened for the same person via `POST /v1/users/{userId}/accounts` skip document verification entirely — the existing approval carries over and the account is booked directly. See [Accounts — Multiple Accounts per User](/platform/accounts).

## Legacy document endpoints

> **Important:** The standalone document upload endpoint (`POST /v1/accounts/{accountId}/documents/upload`) and the W-9 generation endpoint (`POST /v1/documents/w9`) are currently **sandbox stubs** — they accept requests and return success responses, but the submissions are not processed. They are not part of the identity-verification flow. The only document upload that drives KYC is the `documents` array on account creation and resubmission described above.
